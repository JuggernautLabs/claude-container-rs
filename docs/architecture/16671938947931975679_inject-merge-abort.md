# Inject Merge Abort on Conflict

## Problem

`push` runs `inject()` which does `git fetch + git merge` inside the session volume via a throwaway container. If the merge conflicts:

1. `git merge` writes MERGE_HEAD to the repo's `.git/`
2. The script exits with code 1
3. The throwaway container is removed
4. But MERGE_HEAD persists in the volume — it's on disk, not in the container

Result: every subsequent snapshot sees "merge in progress in container" and blocks all operations. The user has no way to fix it without `session exec git merge --abort`.

The error was also misreported as "Docker container wait error" instead of "merge conflict".

## Fix

### Inject script (`src/sync/mod.rs` inject())

Before:
```bash
git merge "_cc_upstream/{branch}" --no-edit || exit 1
```

After:
```bash
if ! git merge "_cc_upstream/{branch}" --no-edit 2>&1; then
    echo "MERGE_CONFLICT"
    git merge --abort 2>/dev/null || true
    git remote remove _cc_upstream 2>/dev/null
    exit 1
fi
```

On conflict: prints `MERGE_CONFLICT` marker, aborts the merge (cleans MERGE_HEAD), removes the upstream remote, then exits. The volume is left in a clean state.

### Error handling

The inject error handler now checks for the `MERGE_CONFLICT` marker in container output and returns `ContainerError::MergeConflict` instead of `InjectionFailed`.

## Files changed

- `src/sync/mod.rs` — inject() script + error handling
