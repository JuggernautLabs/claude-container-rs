# GS-13: Agentic Reconciliation — Merge Into Volume

blocked_by: [GS-6]
unlocks: []

## Problem

When `pull` detects a conflict via trial merge (host-side, in-memory) and the user chooses to reconcile, we launch Claude in a container with CLAUDE.md describing the conflict. But **the conflict doesn't actually exist in the container's repo** — the working tree is clean. Claude has nothing to resolve.

The bash tool's `session_merge_into` solves this by running `git fetch + git merge --no-commit` INSIDE the session volume via a throwaway container BEFORE launching Claude. This leaves real `<<<<<<` conflict markers in the files. Claude then sees them in `git status` and resolves them.

## Current Flow (broken)

```
1. trial_merge detects conflict (host-side, in-memory) ✓
2. User says "resolve" ✓
3. launch_reconciliation:
   a. Stop session container ✓
   b. Write CLAUDE.md with conflict description ✓
   c. Launch reconciliation container ✓
   d. Claude sees clean working tree ✗ ← BUG
```

## Required Flow

```
1. trial_merge detects conflict (host-side, in-memory) ✓
2. User says "resolve" ✓
3. launch_reconciliation:
   a. Stop session container ✓
   b. merge_into_volume: for each conflicted repo:
      - Throwaway container mounts session volume + host repo
      - git fetch /upstream/<branch>
      - git merge upstream/<branch> --no-commit
      - If conflict: leaves <<<<<<< markers in working tree
      - If clean: auto-commits
   c. Write CLAUDE.md + initial prompt ✓
   d. Launch reconciliation container
   e. Claude sees conflict markers in git status ✓
   f. Claude resolves, commits, runs fin
   g. After Claude exits: check .reconcile-complete, re-extract
```

## Scope

### 1. `SyncEngine::merge_into_volume()` — new function

Like `inject()` but uses `--no-commit` to preserve conflicts:

```rust
pub async fn merge_into_volume(
    &self,
    session: &SessionName,
    repo_name: &str,
    host_path: &Path,
    target_branch: &str,
) -> Result<MergeIntoResult, ContainerError>
```

Returns:
```rust
enum MergeIntoResult {
    CleanMerge,           // auto-committed
    Conflict { files: Vec<String> },  // markers left in working tree
    AlreadyUpToDate,
    Error(String),
}
```

Implementation (throwaway container):
```bash
git config --global --add safe.directory "*"
git config --global user.email "claude-container@local"
git config --global user.name "claude-container"
cd "/session/$repo_name"
git remote remove _upstream 2>/dev/null || true
git remote add _upstream "/upstream"
git fetch _upstream "$target_branch"
merge_out=$(git merge "_upstream/$target_branch" --no-edit 2>&1)
merge_rc=$?
git remote remove _upstream 2>/dev/null || true
if echo "$merge_out" | grep -qE "CONFLICT|Automatic merge failed"; then
    echo "CONFLICT"
    git diff --name-only --diff-filter=U 2>/dev/null
elif [ $merge_rc -ne 0 ]; then
    echo "ERROR|$merge_out"
elif echo "$merge_out" | grep -q "Already up to date"; then
    echo "UPTODATE"
else
    echo "MERGED"
fi
```

Key: mounts session volume WRITABLE + host repo READ-ONLY. Runs as host UID (for correct file ownership on conflict markers).

### 2. Update `launch_reconciliation()` in container/mod.rs

Before creating the reconciliation container:
```rust
// Merge target branch INTO session volume repos (creates conflict markers)
for (repo_name, host_path, _) in conflict_repos {
    match engine.merge_into_volume(session_name, repo_name, host_path, "main").await {
        Ok(MergeIntoResult::Conflict { files }) => {
            eprintln!("  ✗ {} — conflict in {} file(s)", repo_name, files.len());
        }
        Ok(MergeIntoResult::CleanMerge) => {
            eprintln!("  ✓ {} — merged cleanly", repo_name);
        }
        Ok(MergeIntoResult::AlreadyUpToDate) => {
            eprintln!("  · {} — already up to date", repo_name);
        }
        Err(e) => {
            eprintln!("  ⚠ {} — {}", repo_name, e);
        }
    }
}
```

### 3. Update CLAUDE.md content

After merge-into, the CLAUDE.md should reflect the ACTUAL state:
- Which repos have conflict markers
- Which repos merged cleanly
- Which repos need manual attention

### 4. Update initial prompt

The initial prompt to Claude should say:
```
I've merged the target branch into your session repos. Some have conflicts:

## hypermemetic/hypno — CONFLICT
  Conflicted files: Dockerfile
  Run `git status` to see the conflict markers (<<<<<<< HEAD)
  After resolving: `git add <files> && git commit`

When all conflicts are resolved, run: fin "<description>"
```

### 5. Host repo mounts

For conflicted repos, mount the host repo read-only at `/host/<repo_name>` so Claude can reference the target branch's version.

For clean-merged repos, no host mount needed.
