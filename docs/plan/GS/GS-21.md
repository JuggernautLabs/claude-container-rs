# GS-21: Volume Migration (5 → 2 + shared)

blocked_by: [GS-18, GS-19]
unlocks: []

## Problem

Existing sessions have 5 volumes each (old layout). New sessions create 2 + shared cache. Need to migrate existing sessions without data loss.

## Scope

### Detection

On startup, scan for old-style volumes:
```
claude-session-{name} exists AND gs-work-{name} does not → needs migration
```

### Migration per session

```
claude-session-{name}  → gs-work-{name}     (copy data)
claude-state-{name}    → gs-state-{name}     (copy data)
claude-cargo-{name}    ─┐
claude-npm-{name}      ─┼→ gs-cache          (merge into shared, mkdir per tool)
claude-pip-{name}      ─┘
```

Implementation:
1. Create target volumes
2. Throwaway container mounts old + new, copies with `cp -a`
3. For cache volumes: copy into `/cache/cargo/`, `/cache/npm/`, `/cache/pip/`
4. Verify target has data
5. Remove old volumes

### Safety

- Never delete old volume until new is verified
- Idempotent: skip if target exists with data
- `--status` flag to preview without acting
- Write `~/.config/git-sandbox/.volumes-migrated` marker
- Per-session: migration can be partial (some sessions migrated, others not)

### CLI

```bash
git-sandbox migrate volumes              # run migration
git-sandbox migrate volumes --status     # show what would be migrated
git-sandbox migrate volumes --force      # re-run even if marker exists
```

### Backward compat

During transition, `discover()` checks for both naming conventions:
```rust
// Check new-style first, then old-style
let work_vol = config.work_volume(name);
let old_work_vol = format!("claude-session-{}", name);
```

Entrypoint handles both mount layouts:
```bash
# Cache is at /cache (new) or /home/developer/.cargo (old)
if [ -d /cache ]; then
    export CARGO_HOME=/cache/cargo
else
    export CARGO_HOME=/home/developer/.cargo
fi
```

### Folded with GS-16

This ticket supersedes the volume migration part of GS-16. The rename (claude-* → gs-*) happens as part of the copy — we rename AND restructure in one pass.

## Files to modify

- `src/migrate.rs` — volume migration logic
- `src/session/mod.rs` — `discover()` checks both naming conventions
- `src/main.rs` — `migrate volumes` subcommand
- `scripts/container/cc-entrypoint` — detect cache mount layout

## Acceptance Criteria

- `migrate volumes --status` shows what would change
- `migrate volumes` copies data and removes old volumes
- Idempotent — running twice does nothing
- Old sessions work during transition (before migration)
- After migration, only 2 + shared volumes per session
