# GS-19: Shared Cache Volume

blocked_by: [GS-18]
unlocks: [GS-21]

## Problem

Each session creates its own cargo/npm/pip volumes. Crates downloaded in session A are re-downloaded in session B. Hundreds of cache volumes accumulate.

## Scope

### One shared cache volume

`{prefix}-cache` mounted at `/cache` in every container. Created on first use (like `ensure_util_image`).

### Entrypoint changes

```bash
export CARGO_HOME=/cache/cargo
export CABAL_DIR=/cache/cabal
export STACK_ROOT=/cache/stack
export NPM_CONFIG_CACHE=/cache/npm
export PIP_CACHE_DIR=/cache/pip
export BUN_INSTALL_CACHE_DIR=/cache/bun
```

### Container creation

`build_create_args` mounts the shared cache:
```rust
args.volumes.push((config.cache_volume(), "/cache".to_string()));
```

Stop creating per-session cargo/npm/pip volumes.

### Session volumes: 2 not 5

```rust
pub struct SessionVolumes {
    pub work: VolumeState,   // repos + config
    pub state: VolumeState,  // .claude conversation
}
```

`create_volumes` creates 2 + ensures shared cache exists.

### Cache volume permissions

Shared cache runs as various UIDs across sessions. Options:
- World-writable (`chmod 777 /cache/*`) — simple, works
- Run as same UID across all sessions (already true — HOST_UID)

## Files to modify

- `src/container/mod.rs` — `build_create_args` mounts cache volume
- `src/lifecycle/mod.rs` — `create_volumes` creates 2 + ensures cache
- `src/types/volume.rs` — `SessionVolumes` shrinks to 2 fields
- `scripts/container/cc-entrypoint` — tool homes point to `/cache/`
- All callers of `cargo_volume()`, `npm_volume()`, `pip_volume()`

## Acceptance Criteria

- New sessions create 2 volumes (not 5)
- `docker volume ls | grep {prefix}` shows `{prefix}-cache` once
- `cargo build` in session B finds crates from session A
- Old 5-volume sessions still work (entrypoint handles both layouts)
