# GS-15: Session Trees — Multi-Container Sessions

## Core Idea

A session is a **tree of containers** sharing volumes. Today a session has one container name hardcoded as `claude-session-ctr-{name}`. This proposal makes sessions naturally multi-container, where each container has a name, role, and configuration — and all share the same session volumes.

## Data Model

```rust
/// A session is volumes + metadata + a tree of containers
struct Session {
    name: SessionName,
    volumes: SessionVolumes,       // the 5 volumes (shared by all containers)
    metadata: SessionMetadata,     // stored on host
    containers: Vec<SessionContainer>,
}

/// One container in the tree
struct SessionContainer {
    /// Unique name within the session (e.g. "work", "reconcile", "experiment")
    label: String,
    /// Docker container name: claude-{label}-ctr-{session}
    container_name: ContainerName,
    /// What this container is for
    role: ContainerRole,
    /// Override image (None = use session default)
    image: Option<ImageRef>,
    /// Extra env vars beyond the session defaults
    env: Vec<(String, String)>,
    /// Extra bind mounts (e.g. host repos read-only for reconcile)
    extra_binds: Vec<String>,
    /// Current state
    state: ContainerState,
}

enum ContainerRole {
    /// Primary work container (the default)
    Work,
    /// Conflict resolution
    Reconcile { target_branch: String },
    /// User-created fork for experimentation
    Fork { description: String },
    /// Headless task runner
    Runner { prompt: String },
}
```

## Naming Convention

```
claude-work-ctr-{session}        ← the default (replaces claude-session-ctr-{session})
claude-reconcile-ctr-{session}   ← conflict resolution
claude-fork-{label}-ctr-{session} ← user-named forks
claude-run-{id}-ctr-{session}    ← headless runs
```

Backward compat: `claude-session-ctr-{session}` is treated as `work` role.

## Volume Sharing

All containers in a session share the same 5 volumes. This is safe because:
- Only ONE container should write to `/workspace` at a time
- Multiple readers are fine (e.g. reconcile reads while work is stopped)
- The `session stop` command stops ALL containers before starting a new one

Unsafe: two containers writing simultaneously. The system should prevent this by default (stop the active container before starting another). A `--force` flag could override for advanced users.

## CLI

### Starting containers

```bash
git-sandbox start -s hypno                  # starts the "work" container (default)
git-sandbox start -s hypno -c reconcile     # starts the "reconcile" container
git-sandbox start -s hypno -c my-experiment # starts a named fork
```

Without `-c`, defaults to "work". If "work" is running and you start "reconcile", it stops "work" first (unless `--force`).

### Forking

```bash
git-sandbox session -s hypno fork experiment "testing new approach"
```

Creates a new container entry with role `Fork`. Same volumes. Can have a different Dockerfile:

```bash
git-sandbox session -s hypno fork gpu-test --dockerfile ./Dockerfile.gpu
```

### Listing

```bash
git-sandbox session -s hypno
  session: hypno
    volumes: 5 (all exist)

    containers:
      ● work         running   claude-work-ctr-hypno
      ○ reconcile    stopped   claude-reconcile-ctr-hypno
      · experiment   no container
```

### Attaching

```bash
git-sandbox start -s hypno -a              # attaches to whichever is running
git-sandbox start -s hypno -a -c reconcile # attaches specifically to reconcile
```

### Stopping

```bash
git-sandbox session -s hypno stop           # stops ALL containers
git-sandbox session -s hypno stop -c work   # stops just the work container
```

### Detach message

```
→ Detached from claude-reconcile-ctr-hypno (reconcile)
  To reattach: git-sandbox start -s hypno -a
  Other containers: work (stopped)
```

## Configuration Storage

Each container's config is stored in the session metadata:

```yaml
# ~/.config/claude-container/sessions/hypno/containers.yml
containers:
  work:
    role: work
    image: claude-dev-hypno
    dockerfile: /path/to/Dockerfile
  reconcile:
    role: reconcile
    target_branch: main
  experiment:
    role: fork
    description: "testing GPU approach"
    image: claude-dev-hypno-gpu
    dockerfile: /path/to/Dockerfile.gpu
```

## Reconciliation Integration

Instead of the current `launch_reconciliation` creating an ad-hoc container, it:

1. Registers a "reconcile" container in the session tree
2. Stops the "work" container
3. Starts the "reconcile" container with the right env/mounts
4. When reconciliation completes (`fin`), removes the "reconcile" entry
5. User can restart "work" normally

## Session set-path

Path management becomes a session-level operation since all containers share volumes:

```bash
git-sandbox session -s veritas2 set-path FormVeritasV2 .
git-sandbox session -s veritas2 set-path FormVeritasV2 /full/path/to/repo
```

Updates `.claude-projects.yml` in the volume. All containers see the change.

## Migration

- `claude-session-ctr-{name}` → recognized as `work` role (backward compat)
- `claude-reconcile-ctr-{name}` → recognized as `reconcile` role
- No containers.yml → auto-create with just "work" entry
- `discover()` scans for both naming conventions

## Implementation Phases

**Phase A (minimal, unblocks daily use):**
- `set-path` command
- `-c` flag on start/stop/attach
- Detach shows container name (done)
- `session show` lists all containers

**Phase B (forking):**
- `session fork` command
- containers.yml storage
- Container role types

**Phase C (reconciliation integration):**
- Reconciliation auto-registers in tree
- Auto-cleanup after `fin`
- One-writer enforcement
