# VM-10: Environment Abstraction — Decouple Git Ops from Transport

## Problem

The `VmBackend` trait mixes two concerns:

1. **Git operations** (ref_read, merge_trees, commit, checkout) — work on any local repo path
2. **Transport** (extract, inject, force_inject) — move data between environments via Docker

This coupling means swapping the environment (Docker → SSH, worktree, cloud) requires reimplementing the entire backend, including the git ops that don't change.

## Design

Split into three traits:

```rust
/// Pure git operations on a local repo path. Universal.
/// One implementation (git2). Never changes.
trait GitOps: Send + Sync {
    async fn ref_read(&self, repo: &Path, ref_name: &str) -> Result<Option<String>>;
    async fn ref_write(&self, repo: &Path, ref_name: &str, hash: &str) -> Result<()>;
    async fn tree_compare(&self, repo: &Path, a: &str, b: &str) -> Result<(bool, u32)>;
    async fn ancestry_check(&self, repo: &Path, a: &str, b: &str) -> Result<AncestryResult>;
    async fn merge_trees(&self, repo: &Path, ours: &str, theirs: &str) -> Result<MergeResult>;
    async fn checkout(&self, repo: &Path, ref_name: &str) -> Result<()>;
    async fn commit(&self, repo: &Path, tree: &str, parents: &[String], msg: &str) -> Result<String>;
}

/// Move git objects between two environments. Per-environment.
trait Transport: Send + Sync {
    /// Transfer a repo's branch from this environment to another.
    /// The implementation decides how (bundle, fetch, rsync, etc).
    async fn transfer(
        &self,
        repo: &str,
        from: &dyn Environment,
        to: &dyn Environment,
        branch: &str,
    ) -> Result<String>;  // returns new HEAD hash on destination

    /// Force-transfer: discard destination state, match source exactly.
    async fn force_transfer(
        &self,
        repo: &str,
        from: &dyn Environment,
        to: &dyn Environment,
        branch: &str,
    ) -> Result<()>;
}

/// An isolated workspace where repos live. Has identity and access method.
trait Environment: Send + Sync {
    fn name(&self) -> &str;

    /// Run a git command inside this environment.
    async fn git(&self, repo: &str, args: &[&str]) -> Result<String>;

    /// Get a path/URL that another environment can fetch from.
    async fn expose(&self, repo: &str) -> Result<String>;

    /// List repos in this environment.
    async fn list_repos(&self) -> Result<Vec<RepoInfo>>;

    /// Read HEAD of a repo in this environment.
    async fn head(&self, repo: &str) -> Result<Option<String>>;
}
```

## Implementations

### Environments

```
DockerEnvironment {
    docker: Docker,
    session_name: String,
    volume: String,
}
- git(): throwaway container with alpine/git
- expose(): mount volume, return /session/repo path
- list_repos(): scanner container (existing SCAN_SCRIPT)
- head(): throwaway container with git rev-parse HEAD

LocalEnvironment {
    base_path: PathBuf,
    repo_paths: BTreeMap<String, PathBuf>,
}
- git(): std::process::Command("git")
- expose(): return the host path directly
- list_repos(): walk base_path for .git dirs
- head(): git2::Repository::open + rev-parse

WorktreeEnvironment {
    base_path: PathBuf,
    worktree_path: PathBuf,
}
- git(): git command in worktree
- expose(): return worktree path
- Same as Local but isolated via git worktree

SshEnvironment {
    host: String,
    user: String,
    base_path: String,
}
- git(): ssh host "cd repo && git args"
- expose(): return ssh://user@host/path
- list_repos(): ssh host "find base_path -name .git"
```

### Transport

```
BundleTransport {
    // Current approach: bundle create + bundle fetch
    // Works for Docker (volume mount), SSH (scp), local (file copy)
}
- transfer(): from.git(repo, ["bundle", "create"]) → to.git(repo, ["fetch", bundle])
- force_transfer(): from.git(repo, ["bundle"]) → to.git(repo, ["fetch"]) → to.git(repo, ["reset", "--hard"])

FetchTransport {
    // Direct git fetch between environments
    // Works when environments can reach each other (local, worktree)
}
- transfer(): exposed = from.expose(repo) → to.git(repo, ["fetch", exposed, branch])
```

## How Ops Change

Current ops that touch transport:
```
Op::Extract { repo, session_branch }     → transfer(container_env, local_env, repo, session_branch)
Op::Inject { repo, branch }             → transfer(local_env, container_env, repo, branch)
Op::ForceInject { repo, branch }         → force_transfer(local_env, container_env, repo, branch)
```

New ops:
```
Op::Transfer { repo, from_env, to_env, branch }
Op::ForceTransfer { repo, from_env, to_env, branch }
```

The VM doesn't know about Docker. It knows about environments. `plan_push` generates `Transfer(local → container)`. `plan_pull` generates `Transfer(container → local)`. The transport implementation handles the how.

## What Stays the Same

- `GitOps` — existing git2_ops.rs, barely changes
- `SyncVM` state model — RepoVM, RefState, BranchName, etc.
- Op tree structure — TryMerge, AgentRun, InteractiveSession
- Program generators — plan_push, plan_pull, plan_sync
- Interpreter — walks ops, checks preconditions, applies postconditions
- StagingBackend — stages ref writes (works with any GitOps impl)
- All tests — just change the backend construction

## Migration Path

### Phase 1: Define traits (no behavior change)
- Add `Environment`, `Transport`, `GitOps` traits to `src/vm/`
- Implement `DockerEnvironment` wrapping existing SyncEngine methods
- Implement `LocalEnvironment` wrapping git2
- Don't change any ops or interpreter yet

### Phase 2: Refactor VmBackend to compose traits
```rust
struct ComposedBackend {
    git: Box<dyn GitOps>,
    transport: Box<dyn Transport>,
    container_env: Box<dyn Environment>,
    local_env: Box<dyn Environment>,
}
impl VmBackend for ComposedBackend { ... }
```
- VmBackend methods delegate to the right trait
- Existing tests unchanged (still call VmBackend)
- New tests verify each trait independently

### Phase 3: Add new environments
- `WorktreeEnvironment` — use `git worktree add` instead of Docker
- `SshEnvironment` — remote repos via SSH
- Test: same program, different environments, same result

### Phase 4: Simplify ops
- Replace `Extract`/`Inject`/`ForceInject` with `Transfer`/`ForceTransfer`
- The environment pair is part of the op, not implicit
- Program generators take `(from_env, to_env)` as parameters

## What This Enables

- `gitvm push` works the same whether the "container" is Docker, a worktree, or SSH
- Agent sessions can run in any environment, not just Docker
- Testing against worktrees instead of Docker (faster, no daemon needed)
- Multi-environment sync: Docker → SSH → Local in one program
- The user picks the environment at session creation, not at sync time
